# Babata 产品需求文档

## 1. 产品定义

Babata 是一个本地优先的个人真实资料系统。它面对的不是单一收藏场景，
而是散落在平台、设备和个人创作中的真实资料从进入、保留、清洗、理解、
沉淀到再次使用的完整过程。

Babata 的价值不在于建立很多协议、接口或宏观架构，而在于让下面这条链路
长期可用，并且尽量低成本、低摩擦：

```text
外部来源 / 自己创作
  -> 收集
  -> 清洗和标准化接入
  -> 消化、吸收、沉淀、建模和管理
  -> 检索、子库、输出、传播和调用
```

产品成功不只意味着“资料已经入库”。资料还必须保留原貌和上下文，能够追溯、
理解、关联和长期管理，并最终能在需要时被找到、组织和使用。

## 2. 用户与使用起点

当前主要用户是 Babata 的拥有者本人。用户通常从以下情境开始：

- 正在飞书、浏览器、内容平台或聊天工具中阅读、收藏或整理资料；
- 已有一批平台导出件、本地文件、书签、媒体或历史记录需要收进来；
- 想写一篇新笔记、修改旧稿、记录反思，或对已有资料作批注和判断；
- 想回看、检索、比较、分析或重新组织已经积累的资料；
- 想把沉淀结果变成子库、报告、卡片、文章、网页或其他应用可用的输出。

日常使用不应要求用户理解内部数据结构，也不应要求手填导出路径、来源元数据或
候选协议。命令行可以服务自动化、恢复和运维，但不是正常收集体验的前提。

## 3. 四段产品边界

| 阶段 | 用户得到什么 | 这一阶段不做什么 |
| --- | --- | --- |
| 收集 | 在来源上下文中发现和选择资料，看到逐条进度，并把原件连同来源上下文可靠拿到 | 不深入理解、不做知识判断、不改写内容 |
| 清洗 | 得到可继续使用的文本、结构、字幕、关键帧和媒体信息，同时保留原件与处理记录 | 不覆盖原件，不把模型解释当成事实 |
| 核心 | 审阅、理解、记录、关联、分类、建模、评分、分析、沉淀并长期管理资料 | 不把自动生成结果静默升级为知识真相 |
| 输出 | 检索、阅读、筛选、回看、组织子库并生成各种可消费结果 | 默认不反向修改权威资料，不成为第二套存储 |

四段必须边界清楚，但对用户仍是一条连续链路。一个入口、适配器或视图不能绕过
这条链路成为隐藏的数据权威。

## 4. 产品行为

### PRD-01：在来源上下文中选择性收集

关联需求：00 §1、§3、§8。

用户连接来源并给出来源、收藏夹、会话、时间段或其他明确范围后，Agent/Babata 应
自己发现候选并继续完成该范围的收集，而不是把逐条确认、翻页和下载交给用户，也不
因为“连接成功”就静默扩大到未授权账号级全量。候选概览应随时可查看，并尽可能显示
标题、来源位置和层级、类型、更新时间、附件可得性及已知限制；只有范围有实质歧义
或会越界时才再次确认。

正常收集应能够从用户已经登录、阅读、收藏或整理资料的真实浏览器上下文开始。当前由
Codex 回收存量时，先直接使用 Codex 的正式 Chrome 浏览器能力完成导航、翻页、候选发现、
结构化读取、网络观察和下载；只有该平台的浏览器控制有真实失败证据，或需要把已经验证
的动作固化成任务结束后仍可调用的重试/重收集命令时，才使用 OpenCLI 并记录理由。长期
路线仍比较官方 API/CLI、`browser-use`、`agent-browser`、Skill、MCP、扩展和 CDP 工具。
站点专用工具负责浏览器层缺少的稳定命令、导出或媒体取得能力；没有合适路径时才增加
窄适配能力。
导出件、PDF、复制、截图和录屏可以作为恢复、离线传递或来源受限时的路径，
但不应被包装成所有来源的日常体验。

微信是当前明确的 UI-only 来源：后续由 Agent 操作官方 PC 微信，在用户一次给定的收藏、
会话或时间范围内完成可见候选、复制、另存和附件下载；不使用进程内存扫描、本地数据库
解密、代理捕获或微信历史 CLI。UI 无法取得的内容显示为受限或失败，不猜测补齐。

浏览器路线必须落到具体平台，而不是把“浏览器”本身当作这些平台的共同来源。知乎、
Bilibili、小红书和抖音要展示真实收藏范围并取得所选正文/媒体；豆包、Kimi 和 ChatGPT
要展示真实会话范围并取得消息顺序、角色和附件；语雀要展示真实知识库/文档层级。
普通当前页、选区、书签和网页收藏仍作为独立浏览器来源，但不能替代这些点名平台的
候选、内容、附件和重收集证据。对浏览器书签，正常产品行为是用户给出一次明确文件夹或
集合范围后，Agent 自动遍历其中的网址并收集实际正文和可得附件；只保存标题与 URL 的
手动剪藏器不算书签来源闭环。当前页/选区快速剪藏可作为未来窄入口，但当前冻结并排在
点名来源与自动存量回收之后。书签自动遍历排到最后，作为独立收集项验收，不与其他来源
扩展混做。

每个来源在进入开发前都必须有真实工具调研：检查并尽可能实际调用官方 API/导出、
CLI、SDK、浏览器插件/用户脚本、MCP/Agent 工具和维护中的开源项目。网页登录来源还
必须先比较通用 Agent 浏览器路线能否复用现有 Chrome 登录态、连续自主运行、下载文件、
观察网络、保存必要会话状态并在阻塞后恢复，再决定是否需要站点专用工具。调研要记录
能取得的内容、层级、附件、历史、重收集能力、最小授权和限制。成熟工具可用时，产品
路线应直接调用或薄包装它，而不是先建设自有协议或适配器。
调查顺序先落到逐来源的官方导出、API/SDK、CLI/MCP 和站点专用工具，再决定通用浏览器
需要补什么。已闭环来源发现更优候选时先登记和验证，不立即替换有效闭环。需要会员的
能力单独记录门槛和收益，等点名来源闭环后统一决策，不把临时购买变成单来源前置。

调查完成后，产品按以下顺序选择当前范围的实际收集路线：

1. 官方免费批量迁移或导出；
2. 现有可用插件或脚本导出；
3. Agent 主导的省心导出；
4. 少量开发后的批量导出；
5. 收费会员/VIP；
6. 重开发或复杂工具流；
7. 持续人机交互配合；
8. 只能手工操作。

前一级能完整、合法地覆盖当前范围时，不无理由使用后一级。使用较低级路线时，用户应能
看到更高级路线不可用的具体原因、当前路线需要自己做什么及会损失什么。VIP 排在重开发
之前只表示先比较购买与开发的成本收益，不代表产品可以自动购买或绕过统一决策。

Agent 已经把明确范围的真实正文、附件和上下文拿回并校验后，当前收集结果可以成立，
不要求先开发平台专用适配器。需要重复执行时优先整理为 Skill；只有实际重复使用证明
现有工具或 Skill 仍缺稳定批量、重试或恢复能力时，才进入少量开发或更重开发。

正常运行应是用户给出一次收集目标或范围，Agent 发起安装、登录或 OAuth 流程；用户
只在来源官方页面完成无法替代的登录、扫码或授权确认。之后的浏览器导航、命令、分页、
下载、重试、临时文件、格式接入和状态由 Agent/Babata 持续完成，直到该范围结束或出现
真正阻塞，不能把中间过程不断交还给用户。产品不得把手抄密码、Cookie、token、导出
路径或来源元数据设计成日常收集步骤。确实只能接收会话秘密的现有工具，必须先证明
没有更安全路线，并把秘密限制在本机受控存储中。

首批需要跑通的真实上下文是飞书文档/Wiki/知识库，以及一个用户已经点名、通过正式
Chrome 登录态访问的具体平台；当前优先完成已有 E2 真实证据的 Kimi 对话。浏览器页面/
书签是独立来源，不作为点名平台已跑通的替代证据。后续来源
是否启用，取决于它能否合法、可靠地给出真实内容、上下文、附件和限制，而不是
取决于是否已经为它预留了名称或空接口。

### PRD-02：收集过程可见、可重试、可重收集

关联需求：00 §3、§7。

一次收集中的每条资料都有明确的 `queued`、`running`、`saved`、`skipped` 或
`failed` 状态。失败信息应能让用户理解问题并重试；局部失败不影响已经成功保存
的资料。

对同一来源再次收集是正常操作。系统应区分 `changed`、`unchanged`、
`inaccessible` 和 `removed`，保留每次收集的时间、结果和来源变化，不用新结果
覆盖旧原件，也不把“暂时访问不到”误判为“资料已删除”。

成功收集的资料应尽可能带有来源链接或导出路径、平台、作者或账号、来源原生
标识、采集时间、来源时间、原始哈希、附件、所在收藏夹/对话/知识库层级、访问
条件和收集工具版本。权限不足、内容不完整或附件缺失必须保持为明确可见的限制。

### PRD-03：原件、第一方资料与派生物分开保留

关联需求：00 §4、§5、§7。

原视频、原图、原音频、原文件、原文和原导出件属于需要长期保护的资料。系统
不得因为已经提取文字、生成摘要或创建视图而丢弃原件。视觉、动态、语气、排版
和上下文本身都可能是内容的一部分。

来源平台生成的预览 PDF、转码媒体、截图或其他转换件不是上传源文件。源文件可取得时，
产品优先保存源文件，并把预览/转换件标成派生物或回退证据；不能用预览下载成功冒充原件
已经拿回。

OCR、转写、字幕、关键帧、视觉描述、摘要、标签、结构化结果和模型理解都是
与输入相连的派生物。它们可以重做、比较或删除后重建，但不能覆盖原件，也不能
伪装成人工确认过的事实。

用户应能从派生结果回到对应原件、输入版本和来源上下文，也能看出一项内容是
外部原件、第一方资料、机器派生结果还是可重建视图。

### PRD-04：清洗独立、忠实且可检查

关联需求：00 §4。

清洗能够处理网页和文本提取、文档解析、PDF/图片 OCR、音视频转写、字幕、
关键帧、视觉描述、媒体元数据、格式统一和基础去重。多模态处理优先支持用户
选择的百炼 CLI，并为后续通义/百炼 API 或批处理保留可替换的处理方式。

每次处理都应记录输入、工具或模型、版本、状态、输出和哈希；失败可理解、可重试，
重跑不会改变原件。用户可以检查原件与派生结果的对应关系，并在资料积累后手动
触发明确范围的批处理。

清洗的目标是让资料可继续使用，而不是把所有内容压成一份“标准文本”。当转换
会损失视觉、时序、语气、版式或上下文时，系统应保留这些限制和回看原件的能力。

### PRD-05：核心区支持理解、沉淀和长期管理

关联需求：00 §1、§2、§5。

核心区是 Babata 的主要价值所在。用户可以审阅收进来的资料，并对其进行理解、
记录、关联、分类、建模、评分、分析、沉淀和长期管理，而不只是浏览一份原始库
或全文搜索结果。P6 的产品定位是一个以用户为中心、高度结构化且动态演进的“个人
知识宇宙”。

用户在工作时应能同时看到相关原件、派生内容、来源、必要历史和已有关系；可以形成
自己的判断、主题、结构和关联，并在确实改写同一作品时保留前后变化。具体交互形态暂不锁定，
但必须支持从“看过一条资料”走到“形成可回顾、可关联、可继续使用的个人沉淀”。

个人知识宇宙至少提供以下产品语义：

- 第一大界是“时间/空间/物质/意识基石 -> 学科 -> 分支”的地图；第二大界是互为理论
  和实践证据的知识与案例；第三大界是长期、中期、短期、实时日志和第一方感悟；
- 学科与分支可演进，内容可跨根、跨学科、跨层级归属；世界观模型或归属实际改变时
  保留必要历史，不把三大界扁平化为同一种记录；
- 每项内容可记录兴趣（现在）、战略（未来）、共识（过去）三个相关度分量和综合分；
  默认权重为 `40/35/25`，权重配置、分值、依据和历史均可回看，且可用真实使用数据
  建议调整；
- 高密度表达可以使用文本脑图、Mermaid、模型、公式、清单、流程或提纲；图片渲染
  是可选的可重建视图，不强迫用户先画图才能沉淀知识。

模型可以提出摘要、标签、关系、分类、评分或分析建议，但这些结果必须标明是
模型输出。未经用户确认，它们不能自动成为事实、人工判断或唯一分类，也不能
覆盖与其冲突的原始证据；但“未审阅”不是阻塞状态，建议仍可进入检索、内容浮现、
关系导航、子库候选和输出候选。用户的接受、修改或拒绝保存为审阅标记；只有用户
实际改写或形成新内容时，才产生新的 first-party C0，而不是把机器记录改名为人工记录。

资料完成收集和清洗后，Agent 应继续提出来源齐全、结构可校验的类型、地图归属、标签、
关系、评分和高密度表达候选，并以机器生成、未审阅的身份进入核心。这个自动消化过程
不要求用户逐条写评论或制造修订；用户没有回应时系统仍继续工作。

### PRD-06：第一方创作走同一条资料链路

关联需求：00 §5。

用户可以新写笔记、草稿和反思，也可以修订已有作品或对其他资料写批注。第一方
创作不是绕过系统的特殊旁路：

- 新写内容形成一条新增资料；
- 只有真正改写同一作品时才形成可追溯的新版本，旧版本仍可回看；
- 批注是独立资料，并与被批注目标建立关系；
- 感悟是第三大界的独立第一方内容，日志是新的时间切片，附件和证据通过资产或关系补充；
- Agent 基于更多资料重新分析时形成新的 C1 处理结果，不冒充第一方作品修订；
- 人工判断与模型生成内容保持可辨别。

用户应能保留真正写作和改写时的原味，同时沿关系看到评论、感悟、日志、证据和后续分析。
版本不是每次普通交互的强制动作；删除任何生成视图都不能造成第一方内容或必要历史丢失。

### PRD-07：资料可检索、回看、关联和组织成子库

关联需求：00 §2、§6。

用户可以按正文、来源、时间、类型、状态、人物、人工分类、关系及处理情况查找
资料，并从结果继续回到原件、来源、版本、派生物和相关资料。

除主动搜索外，系统应根据三维相关度、关系强度、时间、当前方向和用户明确规则，
提供可解释的内容浮现或回顾入口。浮现结果说明“为什么出现”，允许区分人工确认、
模型建议和未审阅候选；它不以黑箱排序替代用户自己的地图和判断。

用户可以为了一个主题、项目、阶段或传播目标，把权威资料组织为分类子库。子库
应保存选择和组织逻辑，必要时可重建；它不是复制出一套需要手工同步的新权威库。

检索和回看不仅覆盖清洗后的文本，也应允许用户发现只有原始媒体、附件、来源
上下文或受限状态的资料，避免“没有可搜索文本”等同于“资料不存在”。

P6 的交付顺序是核心沉淀先行，再完成评分/浮现、检索和关系导航，最后形成子库与
输出；在这些能力产生真实使用价值前，不拆成独立应用。Agent 和 Babata 窄 UI 通过
同一核心工作，Obsidian 只作为可选输出候选。

### PRD-08：输出服务于真实使用，不预设唯一形态

关联需求：00 §6。

用户可以把选定资料和沉淀结果输出为报告、卡片、文章、网页、阅读视图、可分享
内容或供其他应用调用的结果。输出既支持单项使用，也支持明确范围的大规模生成，
并保留必要的来源和版本信息。

输出形态由真实用途逐步确定，不在当前预设为某一个最终应用。Obsidian 可以是
可重建的阅读和展示形式之一，但不是唯一存储，也不要求用户手工维护生成内容。

输出和视图默认只读消费权威资料。删除、重建或更换一个子库、网页、Obsidian 库
或导出物，不应删除原件、第一方内容、派生结果、人工关系或历史版本。

### PRD-09：Skill、脚本和 Agent 是受控入口

关联需求：00 §8、§9。

Skill 可以接收链接、文件和指令，提交给已经存在的收集或处理能力；脚本、浏览器
插件和模型服务也可以生产候选或读取结果。所有入口都应返回真实状态和失败原因，
不能在底层能力尚未实现或验证时声称支持。

这些入口不拥有最终资料，也不能绕过核心直接形成另一条持久化路径。用户触发一次并
给出明确范围后，Agent 应在该范围内连续自主完成收集或批处理，不把逐条确认、翻页、
下载和重试交还给用户。Agent 不因已经连接或技术上可以自动运行，就越过该范围擅自
扩大到账号级全量、自动改写资料或自动作出知识判断。

### PRD-10：本地优先、唯一权威和可恢复

关联需求：00 §7、§9。

Babata 在当前阶段作为一个本地系统工作。真实资料只有一条最终持久化路径；无论
资料来自浏览器、Skill、脚本、模型工具还是第一方创作，都经过同一套版本、溯源
和权威规则。

用户的真实原件、第一方资料、派生结果、视图和运行状态应具有清楚的数据级别：

| 级别 | 内容 | 产品行为 |
| --- | --- | --- |
| C0 | 原始资料、原始媒体、第一方资料及其版本 | 最高优先保护，追加新版本而不原地覆盖 |
| C1 | OCR、转写、摘要、标签、结构化结果和模型输出 | 与输入可追溯，可比较、重做和重建 |
| C2 | Obsidian、网页、子库物化结果和其他生成视图 | 可删除重建，不拥有 C0/C1 |
| C3 | 队列、缓存、日志和其他运行状态 | 可清理，不伪装成业务资料 |

代码、文档、测试和配置模板可以进入 Git；真实资料、数据库、模型输出、日志、凭据
和运行状态必须留在配置的数据根。备份与恢复应保护一致的资料状态，并能验证原件、
版本、关系和索引在恢复后仍然可用。

## 5. 人话词汇表与汇报规则

以下词可以用于内部实现和文档追溯，但用户不需要先学会这些词才能判断进度。产品界面、
进度地图和阶段汇报必须先说明“实际拿回了什么、现在放在哪里、还缺什么”，只有确实有助于
定位时，才在后面用括号补充工程词。不能只说“进入 C0”“有 7 个 asset”或“结果为
unchanged”，却不解释用户实际得到的结果。

| 工程词 | 汇报时先说的人话 | 准确含义 |
| --- | --- | --- |
| C0 | 原件和用户自己写的内容 | 从来源拿回的原文、原图、原音视频、原文件、原导出件，以及用户的第一方创作和历史版本；优先保护，不被加工结果覆盖 |
| C1 | 从原件加工出来的内容 | OCR、转写、字幕、摘要、标签、结构化结果和模型输出；必须能回到输入原件，可以重做 |
| C2 | 随时可以重新生成的阅读或输出页面 | Obsidian、网页、报告、子库物化结果和其他视图；删除视图不等于删除资料 |
| C3 | 任务运行留下的临时状态 | 队列、缓存、日志和运行状态；可以清理，不冒充用户资料 |
| recovery staging | 临时回收区 | 资料已经拿回本机并可供核验，但还可能没有经过 Babata 正式登记；它不是丢失，也不能冒充正式归档完成 |
| Capture | 正式归档 | 把拿回的正文和附件交给 Babata 的唯一保存路径，记录来源、版本、文件指纹和关系，并能从 Babata 回读 |
| Asset | 附件或原文件 | Word、PDF、图片、音频、视频等二进制文件；汇报时应直接说文件类型、数量和是否为原件 |
| Item | 一份资料 | 一段对话、一篇文章、一个视频、一份文档或一条用户创作，是用户可理解的资料单位 |
| Revision | 这份资料的某次保存版本 | 来源内容或用户创作发生变化时追加的新版本；旧版本仍保留 |
| Hash | 文件指纹 | 用文件内容算出的稳定校验值，用于发现损坏、变化和重复；它不是资料标题或用户要填写的编号 |
| Recollect | 再收一次并检查变化 | 对同一明确范围重新读取，判断来源是否变化、暂时不可访问或已删除，不覆盖旧资料 |
| Adapter | 平台专用的长期收集代码 | 当现有 Agent、官方工具、CLI 或 Skill 不能稳定重复完成任务时，才开发的窄连接能力 |
| Skill | Agent 可重复执行的操作流程 | 告诉 Agent 如何使用已验证的浏览器、官方工具或命令完成一类任务；Skill 不是新的资料仓库 |

任务状态也必须翻译成用户能直接采取行动的结果：

| 状态词 | 人话 |
| --- | --- |
| queued | 已列入本次范围，正在等待处理 |
| running | 正在读取、下载或校验 |
| saved | 本次资料已经正式归档并可回读 |
| skipped | 按明确规则没有收，必须说明原因 |
| failed | 没有收好，必须说明失败位置、已保留的成功结果和能否重试 |
| changed | 来源和上次不同，已经或需要保存新版本 |
| unchanged | 来源和上次相同，没有重复保存新版本 |
| inaccessible | 这次暂时无法访问，不能据此判断来源已删除 |
| removed | 有足够证据表明来源已删除；Babata 中的历史原件仍保留 |

“闭环”必须明确说是哪一层，不能用一个词混过三种结果：

1. **Agent 收集闭环**：用户点名范围内的实际正文、附件和上下文已经拿回并校验，缺失和
   限制已经说明。它不要求先开发平台专用适配器。
2. **Babata 正式归档闭环**：拿回的资料已经经过唯一保存路径正式登记，来源、版本、附件
   和文件指纹可回读。资料只在临时回收区时，不能宣称这一层完成。
3. **长期自动化闭环**：同一类任务以后可以由 Agent、Skill、官方工具或必要的窄适配器
   稳定重跑，并正确报告变化、失败和重试。一次 Agent 成功不自动等于这一层完成。

阶段汇报应分别给出这三层的真实状态。某一层未完成时，应直接说明缺的动作及其对用户的
影响；不能用数据库行数、代码文件、测试通过或技术证据等级代替用户可见结果。

## 6. 来源范围

以下是产品期望逐步覆盖的来源范围，不代表已经启用，也不代表必须自建采集器：

| 来源组 | 代表来源 | 期望收集结果 |
| --- | --- | --- |
| 文档与知识库 | 飞书文档/Wiki/知识库、语雀、OneNote、印象笔记 | 正文、层级、附件、作者/时间和来源位置 |
| 聊天与模型对话 | 微信聊天、豆包、Kimi、GPT | 对话顺序、角色、时间、附件和上下文 |
| 内容与收藏平台 | 微信收藏/公众号/视频号、知乎、Bilibili、小红书、抖音 | 原始媒体或可得原件、标题、作者、链接、收藏上下文和限制 |
| 浏览器 | 当前页面、选区、书签和网页收藏 | 页面或选区内容、URL、标题、时间和书签层级 |
| 本地资料 | 文档、PDF、图片、音频、视频和平台导出件 | 原文件、路径/导出上下文、哈希和媒体信息 |
| 第一方创作 | 笔记、草稿、反思、批注和人工判断 | 原始措辞、版本、关系和创作上下文 |

一个来源只有在真实授权路径、内容覆盖、附件、元数据、失败状态和重收集行为经过
验证后，才能显示为“已支持”。无法访问、只能部分收集或依赖手工恢复路径时，产品
必须如实显示能力和限制。

只有网页搜索到工具名或仓库、预留了 provider 文件、写出了导入格式，都不算完成
来源规划。必须同时说明最小授权、Agent 如何调用、实际验证证据和最后回退。

## 7. 当前产品范围与开放问题

当前优先建立完整系统边界，并跑通一个本地的“真实原件进入 -> 可回看”闭环；
飞书和浏览器是首批真实收集路径。阶段顺序、代码骨架、文件清单、接口、命令、
迁移和测试安排由架构与开发流程文档管理，不在本 PRD 中把它们当作产品行为。

当前明确不做：

- 不把产品收窄成收藏器、导入工具、原始资料库或 SQL 查询界面；
- 不为了来源数量静默全量复制，也不先造覆盖所有平台的重型爬虫；
- 不让模型输出覆盖原件、替代人工判断或自动成为知识真相；
- 不要求用户手工维护清洗产物、Obsidian 或其他生成视图；
- 不让命令行、手填路径和手填元数据成为日常收集的主要体验；
- 不先做通用仪表盘、营销式前端或尚无真实用途的外部接口；
- 不因为四段逻辑边界而提前把它们变成多套互相复杂依赖的独立系统。

以下问题保持开放，由真实使用继续决定：

- 核心区最适合的审阅、关联、建模和长期管理交互形态；
- 输出能力最终留在核心中，还是在出现独立使用需求后形成单独应用；
- 飞书和浏览器之后各来源的实际优先级；
- 哪些自动化在资料与规则积累后可以从人工触发升级为人工确认或受控定时运行。
