# Babata 原始需求

## 1. 我真正想做的是什么

Babata 不是为了做一个很严格的协议、很多接口、很宏观的架构。

它要做的是：面对数量繁多的网站、平台和运营商，尽量利用已经成熟的工具，把属于我的真实资料快速拿回来；低成本、低摩擦地收进来，保留它原本的样子，做必要的清洗，然后慢慢消化、沉淀、关联、使用和输出。

它首先是本地优先的个人系统。当前以一个本地应用和一个代码仓库推进；逻辑上有不同部分，但不因为逻辑分层而提前拆成很多独立产品、服务或仓库。

## 2. 整体逻辑

```text
外部来源 / 自己创作
  -> 收集
  -> 清洗和标准化接入
  -> 消化、吸收、沉淀、建模和管理
  -> 检索、子库、输出、传播和调用
```

这四个部分要分清边界，但最终服务于同一件事：真实资料能被收进来、留住、看懂、找得到、用起来。

## 3. 收集：只负责“拿到东西”

收集部分是一个收集者，类似爬虫、脚本、浏览器插件、官方导出和现有工具方法的聚合体。

它只负责把东西拿到，不承担深入理解、知识判断、建模或内容改写。优先使用已有方案；确实没有现成路径，才针对性开发很窄的适配器。

系统建设早期可以先利用正在运行的 Codex 已有能力、官方连接器、正式版 Chrome 登录态和必要的桌面操作，在一个具体平台上真实跑通存量读取，确认 Agent 确实能自主导航、翻页、读取正文、识别附件路径和限制，并把明确范围的资料拿回本机。路线一旦用真实资料证明可行，就继续当前系统阶段，不让长期自动化准备或无限扩大的存量回收阻塞基础建设。长期自动收集和脱离 Codex 独立运行仍然需要；以后可以由 Skill 触发 Browser Use、Agent Browser 或同类本地工具，但 Codex、Chrome 和这些浏览器工具都只是实现省心收集的当前手段，不是产品目的。

当 Agent 已经把明确范围的真实正文、附件和上下文拿回并校验，当前收集目标就可以成立，
不自动升级成平台专用适配器开发。以后需要重复执行时，优先把已验证流程整理成 Skill；
只有反复使用证明现有工具或 Skill 仍缺稳定性、批量、重试或恢复能力时，才进入少量开发
或更重开发。

这里的“优先使用已有方案”不是列几个可能的工具名，也不是先建一批 adapter 空文件。对每一个要支持的来源，都要先真实调查并尽可能实际调用：官方 API 和官方导出、成熟 CLI/SDK、浏览器插件和用户脚本、MCP/Agent 可调用工具、仍在维护的开源项目分别能拿到什么，需要什么授权，能否保留层级、附件、时间和上下文，能否重复获取，以及有什么明确限制。已经有成熟工具能完成的，就直接使用或只包一层很薄的调用，不重新造一套。

逐来源的顺序必须具体执行为：先检查官方导出、官方 API/SDK、官方 CLI/MCP，再检查仍在维护的站点专用 CLI、扩展和开源工具，最后才判断通用浏览器路线缺什么。已经真实闭环且效果正常的来源也要补做这项路线审计，但只记录更优候选，不在未验证前改动现有闭环。某项官方 API、MCP、导出格式或批处理能力如果要求会员，先记录能力、门槛和影响，不在单一来源处理中购买或启用；等点名来源都闭环后再统一决策。

工具调查完成后，收集器选择实际执行路线时按以下顺序逐级判断，前一级能完整覆盖当前
范围就不无理由降级：

1. 官方免费批量迁移或导出；
2. 现有可用插件或脚本导出；
3. Agent 主导的省心导出，用户只完成无法替代的登录、授权或范围确认；
4. 少量开发后实现的批量导出；
5. 收费会员/VIP 能力；
6. 重开发或复杂工具流；
7. 需要持续人机交互配合的路线；
8. 只能由用户手工操作的路线。

这八级是调查后的实际路线优先级，不取代逐来源工具调查。选择较低级路线时，要说明
更高级路线为什么不可用、缺什么或为什么不适合当前范围。收费 VIP 排在重开发之前表示
应先比较费用、覆盖和省心程度，不代表系统可以自行购买；仍需记录门槛，并由用户统一
决策或明确授权。

数量繁多、主要存在于登录网页里的来源，不应一开始就逐站寻找或编写爬虫。应先到 GitHub 真实调查能被现有 Agent 直接调用的通用浏览器导航层，优先看成熟的 Agent Browser、browser-use 一类 CLI、Skill、MCP 或 CDP 工具，验证它们能否复用我已经登录的真实浏览器，由 Agent 自己完成导航、翻页、候选发现、结构化读取、网络观察、下载、重试和异常恢复。站点专用 CLI、下载器和导出工具仍然有价值，但应补充通用浏览器层确实缺少的稳定结构或媒体取得能力，而不是把每个网站都重新做一遍。

浏览器只是取得网页登录资料的手段，不是一个可以替代具体来源的含糊来源名。对我已经
点名的知乎、Bilibili、小红书、抖音收藏，以及豆包、Kimi、GPT 对话和语雀，必须分别
展示它们自己的收藏夹、会话、知识库、时间或数量范围，拿到该平台实际的正文、媒体、
附件和上下文，并按具体来源记录限制与重收集结果。只做一个能提交任意当前页、选区或
候选 JSON 的通用浏览器壳，不能代表这些来源已经开始收集或已经支持。浏览器书签和普通
网页仍是独立来源，但不能拿它们替代上述平台。当前书签收集的目标也不是让用户逐条点击
一个剪藏器只保存标题和 URL；用户给出明确书签范围后，应由 Agent 自动遍历书签网址，
取得实际正文和可得附件，再进入同一 C0 链路。手动当前页/选区剪藏器只保留为未来窄入口，
冻结并排在点名来源、存量回收和自动书签正文路径之后，不作为当前阶段完成条件。

对于已连接的来源，我给出要收集的来源、收藏夹、会话、时间段或其他明确范围后，Agent/Babata 应自己发现候选并继续完成收集；候选概览、标题、来源位置、层级、类型、更新时间、附件可得性和已知限制应随时可查看，但不能把逐条点击确认变成我的日常工作。只有范围确实有歧义、会扩大到未授权账号级全量，或平台要求不可替代的登录/授权动作时才停下来问我。不要因为已经连接就静默复制未授权范围。

收集应有每条资料的 queued、running、saved、skipped、failed 状态。失败必须可理解、可重试，不能让已经成功的资料丢失。重收集是正常动作：应分清 changed、unchanged、inaccessible 和 removed，并追加溯源，而不是覆盖原始资料。

首批候选来源包括但不限于：

- 微信收藏、公众号、视频号、聊天记录；
- 飞书文档、Wiki、知识库、云文档；
- 语雀；
- OneNote、印象笔记；
- 豆包、Kimi、GPT 聊天记录；
- Bilibili、抖音、小红书、知乎收藏；
- 浏览器书签、网页收藏；
- 本地文件；
- 我自己写的内容。

当前存量回收中，OneNote 直接使用官方整本 PDF/MHT 导出；印象笔记优先使用官方整库
`.notes` 导出，由 Agent 使用已经公开且在真实导出件上校验成功的固定算法解密正文，同时
保留原 `.notes`。网页版 DOM 直接读取和客户端单篇 MHT 只作为补充或回退，不再要求用户
逐篇手工导出 HTML。抖音和视频号暂时不处理；优先级变化不删除这两个来源，也不把未跑过
的样本或未形成的自动化写成已经支持，后续只有用户重新启用时才继续。

微信后续只走官方 PC 微信 UI，由 Agent 在用户给出一次会话、收藏分类、时间或数量范围后
带着完成可见候选、复制、另存和附件下载；不再把进程内存扫描、本地数据库解密、代理捕获
或微信历史 CLI 列为待执行路线。浏览器书签不急，排在其他点名来源之后，最后作为一个
独立收集项处理；届时仍应由 Agent 自动遍历网址并取得正文和可得附件，而不是逐条手工剪藏。

来源路径以“合法且最少摩擦”为原则：先用已经验证可用、能够被人或 Agent 调用的官方 API、CLI、SDK、浏览器扩展/用户脚本、MCP 或成熟开源工具；需要时只增加很薄的调用和格式接入。现有工具确实缺少关键能力时，才开发窄适配器。官方一次性整本/整库导出属于第一级正常路线；逐页手工 PDF、复制、截图和录屏才是所有正常工具路线都走不通后的最后回退，不能为了方便本地测试而冒充主路径。

平台为了预览或阅读而生成的 PDF、转码文件、截图或其他转换件，不能冒充用户原先上传、
保存或创作的 Word、图片、音视频等源文件。源文件实际可取得时必须优先拿回源文件；预览
或转换件只能作为派生物、补充证据或明确受限的回退结果。

当前由 Codex 回收各平台存量资料时，先按八级路线检查官方免费批量迁移/导出和现有可用
插件/脚本。确定进入第三级 Agent 省心导出后，优先直接使用 Codex 自带的正式 Chrome
浏览器能力探索真实候选、页面和网络响应。OpenCLI 不是默认绕行路线；只有浏览器控制在
该平台被真实证明不稳定，或者需要把已经验证的取得动作变成任务结束后仍可调用的重试/
重收集命令时，才使用 OpenCLI，并记录具体理由。无论使用哪种手段，都必须回到同一条
Babata 正式保存路径。

正常情况下，我只需要给出一次收集目标或范围，并完成来源本身无法替代的最小动作：在官方页面登录或确认 OAuth、扫码、批准必要的本机/浏览器权限。之后的工具安装、浏览器导航、候选发现、命令调用、分页、下载、重试、临时目录、格式接入、状态和溯源都由 Agent/Babata 处理，直到范围完成或遇到真正无法继续的阻塞；不要把过程不断交还给我操心。不要把密码、Cookie、token、导出路径或元数据手抄给 Agent 当成日常流程；只有某个现有工具被真实证明没有更安全的授权方式时，才把受控秘密导入作为明确限制和最后选择。

每次收集都应尽量留下来源链接或导出路径、平台、作者/账号、采集时间、原始哈希、附件、所在收藏夹/对话/知识库层级、访问条件和收集工具版本。拿到一段孤立文本不算完成。

权限不足、来源受限、资料不完整或收集失败的情况不能假装收好了；必须是明确可见的受限、失败或待处理状态。

进度和结果必须先用人话分别说明：资料是否已经由 Agent 拿回并校验、是否已经正式登记进
Babata、是否已经能够长期重复自动执行。三种结果不能混说，也不能只用 C0、asset、
revision 或测试通过等工程词代替用户实际得到什么、放在哪里和还缺什么。

## 4. 清洗和标准化接入

清洗要单独拎出来，不和收集混在一起。

收集拿到的是原件、导出件、链接、截图、录屏、网页和文件。清洗负责把它们变成系统可以继续使用的资料，例如：

- 网页和文本提取；
- 文档解析；
- PDF、图片 OCR；
- 音频、视频转文字；
- 字幕、关键帧、视觉描述；
- 媒体元数据；
- 格式统一和基础去重。

我不追求一分钱不花。多模态清洗优先考虑百炼 CLI，或者通义/百炼 API；付费解决是可以接受的。

但清洗要尽可能保持原味。

原视频、原图、原音频、原文件、原文、原导出件都可以留着，很多情况下应该留着。很多资料的价值本来就在视觉、动态、语气、排版和上下文里，不能因为转成文字或生成了摘要，就把原来的东西丢掉。

OCR、转写、摘要、标签、结构化结果和模型理解都是派生物，不是原件，也不能覆盖原件。每一步清洗都应能追溯输入、工具或模型、版本、处理状态、输出和哈希。

## 5. 消化、吸收、沉淀、运用、分析、记录、建模和管理

这是原 Compass 最核心的部分，也是 Babata 最终真正要承担的部分。

它负责让我对收进来的资料做审阅、理解、记录、关联、分类、建模、评分、分析、沉淀和长期管理。它不是“给清洗结果分几个文件夹”，而是逐步形成一个以“我”为中心、高度结构化且动态演进的个人知识宇宙。

这个个人知识宇宙保留以下结构：

- 动态相关度按“兴趣/现在、战略/未来、共识/过去”三个维度表达；默认综合权重为
  `40/35/25`，但权重、评分依据和调整历史必须版本化，并允许以后根据真实使用数据调节；
- 宏观地图采用“基石 -> 学科 -> 分支”三级结构；顶层固定为时间、空间、物质、意识
  四个世界观基石，学科和分支可以继续生长；内容可以同时归属多个基石、学科或分支；
  如以后改变四个顶层基石，应作为明确、可追溯的世界观模型版本，而不是静默改名；
- 正式语义至少包括地图/方向、知识、案例、日志和感悟。知识表达经过消化的理论，案例
  表达现实中的实践标本，二者可以相互印证；日志保留长期、中期、短期和实时等时间切片；
  感悟保留我自己形成的灵感、框架和成熟思考；
- 统一标签和双向关系贯穿地图、知识、案例、日志和感悟，允许跨学科、跨层级、多重归属，
  不把树形目录当成唯一关系；
- 高密度表达是正式内容的一部分。优先保留 Markdown、Mermaid 或其他文本可重建的脑图，
  以及模型、公式、清单、流程和提纲；渲染成脑图图片是可选、可重建的展示结果，不是强制
  写作步骤，也不是唯一权威；
- 系统既要支持主动检索，也要根据相关度、关系、时间和当前方向让值得回看的内容自然浮现，
  并允许沿版本、来源、标签和双向关系继续探索。

模型可以帮忙，但模型输出不是事实，不应自动变成知识真相，更不能覆盖原始资料。

模型提出的标签、关系、分类、评分、摘要或分析建议，在没有人工决定时也可以继续用于检索、
关系导航、内容浮现、子库候选和输出候选；但必须始终标明模型、版本、依据和“未审阅”状态，
不能冒充我的判断或已确认事实。我的接受、修改、拒绝只是可追溯的审阅标记，不是阻塞整个
系统的审批流；修改建议形成了我的新内容时，再作为 first-party 新版本保存。

我自己手写的内容不是一个特殊旁路。

我写的新笔记、草稿、反思、批注和人工判断，和飞书、Bilibili、知乎里的内容一样，都是一种来源，只不过这个来源是 first-party。

- 我新写一篇，是新增资料；
- 我修改一篇，是一个新版本；
- 我对别的资料写批注，是一条独立资料，再和目标建立关系；
- 不直接把旧内容原地改掉。

这样既能保留我当时写东西的原味，也能保留后续的变化和判断。

P6 的实现顺序应是：先让核心知识沉淀成立，再建立相关度、内容浮现、检索和关系导航，最后
形成子库和输出。不要先做一个 Obsidian 库、展示站或独立应用来代替核心。

## 6. 输出、子库、分类、传播和调用

Babata 最后要能随时大规模地把沉淀过的东西用出来。

可以是检索、阅读、筛选和回看；可以是分类子库；可以是报告、卡片、文章、网页；可以给别的应用调用；也可以是各种传播和输出形式。

这一部分暂时不预设最终产品形态。它可以以后成为独立应用，也可以继续放在核心里。先保证资料是真的、完整的、可追溯的、可消化的；输出形态按真实使用价值长出来。

输出和视图默认不反向修改核心资料。删除一个 Obsidian 库、一个网页视图或一个导出物，不应该导致原始资料、清洗结果或我的手写内容丢失。

Obsidian 可以有，但它只是展示和阅读形式之一，不是唯一存储，也不应该要求手工直接维护。清洗后的数据、Obsidian 数据和各种展示页面，尽量都应从权威资料重建。

P6 当前仍由 Babata 核心提供唯一正式写入，Agent 和窄 UI 是主要交互候选，Obsidian、网页、
报告和其他应用只是输出候选。旧 1.0 的代码、算法和文档只在确实有用时作为参考，不整体
迁移旧架构或旧数据，也不因为保留原始意图而恢复旧系统的技术决定。

## 7. 数据、版本和权威

代码、文档、测试和配置模板进入 Git。

真实原件、SQLite、清洗产物、缓存、日志、凭据和运行状态不和代码 Git 混在一起，放在外部数据根，再通过一致性快照、加密增量备份、NAS/云端副本和恢复验证来保护。

原始资料和第一方资料是最高优先级的 C0 数据；派生资料是 C1；可重建视图是 C2；运行状态和日志是 C3。

原始资料应尽量 append-only。派生资料单独记录，可重建；视图可删除重建；任何下游展示、脚本、插件或模型服务都不能成为另一个隐藏的权威写入者。

## 8. 交互、Skill 和自动化

Skill、脚本、浏览器插件和模型服务都只是入口、工具或候选资料生产者，不是数据权威。

Skill 可以接收链接、文件或指令，交给收集链路；也可以调用已经验证过的本地能力。但 Skill 不应绕过核心直接写资料，也不应在没有真实命令和测试时假装支持某项能力。

Agent 自动化首先服务于“我触发一次，之后不用操心”：我给出来源或明确范围后，Agent 应尽可能把收集跑完，只在不可替代的登录、授权、范围歧义或高风险动作处找我。没有我的来源/范围指令，不静默扩大到账号级全量；但已经明确授权的范围内，不要用反复人工确认妨碍自动收集。收集自动化不等于自动做知识判断或擅自改写资料。

命令行是自动化、恢复和运维入口；它不应该成为我日常收集资料时必须手动输入导出路径、元数据或导入命令的主要体验。需要时应有很窄的收集器 UI 或浏览器扩展，通过本地受保护接口调用同一条核心链路。

## 9. 技术边界

Babata 当前是一个本地优先的系统，不急着拆成很多独立仓库、服务、协议或对外 API。

Rust 尽量承担核心：领域、存储、迁移、处理、来源适配、CLI、worker 和备份。JavaScript/TypeScript 只在浏览器插件等必须在浏览器运行的地方使用。Python 只在有成熟 Python 工具确实无法替代时使用，而且只能作为受控边界，不能直接写核心数据。

最终持久化真实资料的路径必须统一。浏览器扩展、脚本、Skill、Python 工具、模型服务和视图只能提交候选资料、调用本地能力或读取结果，不能绕过核心直接写数据库或落最终原件。

## 10. 不要做什么

- 不回到五仓库、多协议、先接口化再找用途的架构；
- 不为了宏观、通用、未来可能性而忘记真实目的；
- 不做万能爬虫、绕过访问控制或把未经验证的第三方工具当成已支持；
- 不在没有逐来源真实工具调研和调用证据时，先用接口、协议、空 adapter 或手工导出替代收集方案；
- 不让模型输出覆盖原件或自动成为知识真相；
- 不把手工导出、手填路径、手填元数据和命令行导入变成正常收集路径；
- 不做营销式前端；
- 不因为逻辑上有收集、清洗、核心和输出，就提前把它们拆成互相复杂依赖的独立系统。
