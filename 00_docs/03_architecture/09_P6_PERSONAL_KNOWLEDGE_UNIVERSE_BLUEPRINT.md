## 1. 文档职责与状态

本文把 Babata 1.0 原始归档中有关个人知识宇宙的意图，连同 2026-07-20 已确认的九项
决定，翻译为 P6 的产品与架构基线。它继承 `00_REQUIREMENTS.md`、`01_PRD.md`、
`02_ACCEPTANCE_CRITERIA.md` 和 `03_ARCHITECTURE.md`，不能覆盖上游决定。

本文是设计基线，不是实现证据。2026-07-20，P6.1 已启动首个纵向切片；只有对应验收
标准和测试用例用真实资料通过后，相关 P6 子阶段才可完成。

## 2. P6 的产品定义

P6 不是简单地给 C1 清洗结果分目录，也不是先做一个全文搜索页面。它要把外部原件、
第一方创作和清洗派生物转化为一个以用户为中心、高度结构化且动态演进的个人知识宇宙：

```text
C0 原件 / 第一方版本 + C1 清洗结果 / 模型建议
  -> Agent 消化、分类、建模、关联和质量校验
  -> 未审阅机器候选进入核心，不等待用户逐条决定
  -> 三大界：地图架构 / 知识与案例 / 日志与感悟
  -> 相关度评分与可解释浮现
  -> 检索和关系导航
  -> 子库和输出
```

核心先成立，检索与浮现随后，子库和输出最后。P6 暂不拆独立应用。

## 3. 知识宇宙的语义结构

### 3.1 三级地图

地图使用稳定的三级语义，而不是只能单选的文件夹树：

```text
基石（世界观维度）
  -> 学科（领域维度）
     -> 分支（专业维度）
```

顶层固定为四个稳定根：

| 根 | 覆盖示例 |
| --- | --- |
| 时间 | 时间、历史、科学史、哲学史 |
| 空间 | 地理、坐标系统、宇宙学 |
| 物质 | 宏观宇宙、恒星、微观粒子、肉身与生物学 |
| 意识 | 心理学、认知科学、自我、思维模型、模拟理论 |

学科和分支不是固定枚举，可以随认知发展创建、改名、合并、停用或建立关系。一个学科、
分支或内容可以归属多个上层节点，例如博弈论可以同时关联数学、社会学和意识。归属关系
有来源，实际改变时保留必要历史，不通过复制内容实现多重归属。

四根是当前已确认的 P6 世界观模型，不等同于对“物质是否存在、宇宙是否存在、人是否
存在、虚无是否存在”等哲学问题的回答。若未来改变四根，必须创建新的 WorldviewMap
版本、说明原因并显式迁移归属，旧版本继续可读。

### 3.2 三大界与正式语义类型

| 界 | 类型 | 作用 | 最小语义 |
| --- | --- | --- | --- |
| 第一大界：知识宇宙地图 | Map/Direction | 表达世界观、学科、分支和当前方向 | 层级、关系、状态、变化依据 |
| 第二大界：知识的原子与标本 | Knowledge | 经过消化和浓缩的理论单元 | 标题、正文、来源/依据、关系 |
| 第二大界：知识的原子与标本 | Case | 理论在现实中的实践标本 | 场景、过程、结果、分析、关系 |
| 第三大界：我的认知轨迹 | Log | 按时间和重要程度保存认知轨迹 | 时间尺度、发生时间、正文、关系 |
| 第三大界：我的认知轨迹 | Insight | 用户与知识互动后形成的原创思想 | 作者、成熟度、正文、依据、关系 |

五类语义不能脱离三大界被当成平铺枚举。外部原件和 C1 清洗结果也不因存在于数据库中
自动成为 Knowledge 或 Case；它们是 Agent 消化、语义候选和关系建模的输入。

日志的时间尺度至少支持：

- 长期：方向性、里程碑式的重要节点；
- 中期：经过压缩和概括的阶段总结；
- 短期：重要片段，少压缩并保留原始信息；
- 实时：临时、待整理的原始记录。

实时日志不是外部资料的替代入口。它是新的 first-party C0 时间切片；以后整理成知识、
案例或感悟时，通过关系连接，不原地抹掉实时记录，也不把整理结果称为日志 `v2`。

### 3.3 全局关系

统一标签和显式关系同时存在：标签适合宽松聚合，关系表达具体语义。至少支持：

- 知识由案例佐证、案例应用或反驳知识；
- 日志为感悟或判断提供背景；
- 感悟引用知识作为论据；
- 内容归属一个或多个地图节点；
- 内容引用 C0 原件、C1 派生物、旧版本或其他 first-party 内容；
- 人工关系与模型建议关系保持来源可辨别。

关系是可追溯记录，不把“反向链接”只留给某个展示工具临时计算。删除 C2 视图后，关系
仍完整存在。

### 3.4 新内容、关系、再分析和真实修订

P6 必须根据动作的真实含义写入，不能统一折叠成线性版本：

| 动作 | 正式结果 |
| --- | --- |
| 对资料评论或批注 | 新的 first-party 资料，并关联目标当时的 item/revision |
| 形成灵感、框架或成熟思考 | 第三大界 Insight，可引用资料、知识、案例或日志 |
| 记录新事件或阶段状态 | 第三大界新的 Log 时间切片 |
| 增加附件或证据 | 资产、证据或关系；除非正文也被真实改写，否则不是内容修订 |
| Agent 基于更多输入重新消化 | 新的 C1 run/suggestion，保留输入、模型、依据和取代/补充关系 |
| 真正改写同一第一方作品 | 新的 first-party revision，旧措辞和附件保持可读 |
| 世界观地图或评分 profile 改变 | 新配置/模型历史，旧归属和旧计算结果可复盘 |

Git、NAS、快照和备份承担同步、容灾和时间点恢复；它们不替代上述来源/处理溯源，也不构成
给每条内容强制制造 `v1/v2/v3` 的理由。

## 4. 动态相关度评分

### 4.1 三个维度

任何地图节点或内容都可以有三个独立分量：

| 维度 | 时间取向 | 含义 |
| --- | --- | --- |
| interest | 现在 | 当前热情与正在投入时间精力的程度 |
| strategy | 未来 | 长期目标、战略布局和潜在发展方向的重要程度 |
| consensus | 过去 | 已被验证、基础性、构成认知基石的程度 |

首个默认权重 profile 为：

```text
relevance = 0.40 * interest + 0.35 * strategy + 0.25 * consensus
```

具体分值范围在实现前由测试夹具和 UI 精度共同确定；不能因为范围尚未锁定而改变三维
含义或默认权重。

### 4.2 版本与真实数据调节

每次评分至少保留对象、三个原始分量、综合分、依据、作者/模型、时间和所用 profile
版本。历史评分不因重新计算而覆盖。

真实使用数据可以包括查看、检索命中后的继续阅读、主动收藏、建立关系、加入子库、引用
到输出和明确降权等行为。它们用于产生调分或调权建议，不直接把行为次数当作事实，也不
在后台静默改变 profile。用户未处理调权建议时，当前 profile 继续工作，不阻塞任何流程；
采用新权重时创建新 profile，旧 profile 和旧结果仍可复盘。

## 5. 高密度表达

知识和分析应支持把最精华部分表达为：

- Markdown 或缩进文本脑图；
- Mermaid 图；
- 模型、公式和变量说明；
- 清单、步骤、流程、决策树和提纲；
- 对高密度表达的详细正文解释。

这类文本是可搜索、可 diff、可引用，并在真实改写时可保留历史的正式内容。适合时由 Agent 顺手提出或
生成文本脑图，但不是每条内容的审批门槛。PNG/SVG 等渲染图属于可选 C2；删除图片后，
文本表达仍可用并可重新渲染。

## 6. AI 建议不是审批流

完成收集和 C1 清洗只代表资料可被继续处理，不代表它已经成为第二大界的 Knowledge。
Agent 还应提出一个可校验的语义候选包：候选类型及所属大界、地图归属、标签、关系、三维
评分、高密度表达、输入证据、模型/provider 和限制。通过来源、结构和质量检查后，这个包
以 C1 机器身份进入 P6 核心并可被检索、浮现和继续关联，不等待用户逐条回复。

`ModelSuggestion` 保持 C1 机器身份，至少记录 suggestion kind、模型/provider、版本、
输入证据、输出、时间和状态。审阅状态至少可表达：

```text
unreviewed -> accepted | modified | rejected
```

`SuggestionReview` 是用户留下的 first-party 状态标记，不改变建议的作者和类型：

- `unreviewed` 建议可用于检索、浮现、关系导航、子库候选和输出候选；
- `accepted` 只表示用户认可该建议，不复制一份内容冒充人工原创；
- `modified` 在用户确实形成新内容时创建相应类型的 first-party C0 并引用原建议；只有
  明确改写既有第一方作品时才追加该作品的 revision；
- `rejected` 保留建议及理由，不让它继续作为默认候选，但不删除历史；
- 没有任何审阅决定时，系统照常工作，并在所有消费面保持“机器、未审阅”标识。

人工确认、机器建议和计算结果在搜索、浮现、关系、子库与输出中都必须可区分。下游可以
按用途纳入或排除未审阅建议，但不能全局假定“未审阅等于不可用”。

## 7. 检索、浮现与关系导航

P6 同时提供主动和被动两种发现方式：

```text
主动：查询 / 筛选 / 排序 -> 结果 -> 原件、版本、关系和上下文
被动：当前方向 + 三维相关度 + 时间 + 关系 -> 可解释浮现 -> 继续探索或忽略
```

检索覆盖正文、来源、时间、语义类型、状态、人物、地图归属、标签、关系、处理状态、
三维分量、综合分和 profile 版本。只有媒体、附件或受限状态的资料仍可按 C0 元数据与关系
发现。

浮现不是另一个信息流产品，也不是黑箱推荐权威。每个结果至少显示一项主要原因，例如
“当前战略方向”“与刚查看的知识有佐证关系”“长期未回看且共识分高”。忽略浮现不产生
强制待办；模型候选和未审阅关系必须显式标识。

## 8. 交互与写入权威

P6 的交互遵循当前 Babata 状态：

```text
Agent / Babata 窄 UI
  -> application use case
  -> Babata 核心校验和唯一正式写入
  -> C0/C1 + 可重建读投影
  -> Obsidian / Web / 报告 / 其他 C2 输出
```

Agent 适合批量分类建议、关系建议、高密度表达草案和自然语言查询；窄 UI 适合连续审阅、
对照原件、快速修订、标记和关系导航。两者调用同一核心能力，不各自直接改 SQLite 或正式
文件。没有界面的能力可以通过 Agent/CLI 验收；具体 UI 尚不存在时，所有 UI 视觉和交互
细节要求可忽略，但数据语义、状态和权限边界不能忽略。

Obsidian 只是可选的阅读/展示 builder。对 Obsidian 文件的外部编辑默认不回写；若未来
确有回写需求，按新的 first-party 输入重新进入核心。

## 9. 子库与输出

`SublibraryDefinition` 是版本化 first-party C0，保存用途、范围、规则、人工纳入/排除和
组织方式。其物化目录、搜索投影和展示文件是 C2，可删除重建。

输出读取明确范围内的 C0、C1 和人工沉淀，并生成 manifest，至少记录输入版本、builder/
template/profile 版本、时间、状态、限制和回溯路径。输出可以选择是否包含未审阅建议；
包含时必须保留标识。报告、卡片、文章、网页、Obsidian 和供其他应用消费的结构化结果
都是 builder 类型，不是新权威。

## 10. 权威与存储映射

语义类型不单独决定数据级别，作者和可重建性才决定：

| 记录 | 权威级别 |
| --- | --- |

## 14. 仍待实现时决定的事项

九项产品问题已经足够闭环，不再阻塞 P6 设计。Issue #65 已由真实资料锁定以下 P6.1
实现选择：

- 三个分量均使用 `0..100` 整数；综合分按 profile 权重计算并以 `0..10000` 整数保存，
  避免浮点漂移；展示层可以除以 100；
- P6 核心作为现有 `raw.sqlite` 的受迁移扩展，继续由 Rust application/core 和
  `SqliteRawRepository` 形成唯一正式事务写入，不另建 P6 数据库；C1 原始结构化输出仍在
  `derived.sqlite`，核心规范化记录必须引用 derivative ID 与 output hash；
- 机器候选以 `p6-semantic-candidate/v1` JSON 合同进入核心，一次事务同时落 suggestion、
  三级地图节点/边、语义条目、多重归属、标签、关系、高密度表达和评分；
- 首个真实自动消化 provider 为已鉴权百炼 CLI 的 `qwen-plus`，prompt/version、C0 revision、
  C1 evidence ID/hash、task/usage、输出 hash、限制和未审阅身份均保留；
- first-party Log/Insight 必须先形成 ready C0，且登记正文必须与该 C0 原文完全一致；核心
  只保存语义登记和关系，不允许 JSON/CLI 另造一份“用户原话”。
- 补充原件、平台预览或其他证据使用独立 `asset_attachment_operations` 追加到既有 ready
  revision；操作保存 reason、metadata、状态、失败和 asset membership，不分配正文 ordinal，
  失败只隔离本次操作及其附件。`workspace revise` 对正文 hash 未变化的请求直接拒绝；
- 学科和分支使用稳定节点身份与当前生命周期；新增、改名、停用、合并、父边变化、内容
  归属变化和地图标签变化均追加事件。合并只迁移当前父边、子边、内容归属和标签，不删除
  旧节点、旧评分或历史；P6 baseline 的四基石由应用与数据库共同拒绝静默修改；
- 相关度评分目标统一支持语义内容和地图节点；每次评分读回 profile、三个分量、综合分、
  依据、作者身份和时间。新 profile 追加而不覆盖旧 profile 或旧评分；
- P6.1 只启用一项窄 C2：从核心高密度文本生成带 source/output hash 和 manifest 的 Markdown
  预览，支持校验、删除和重建，不反写 C0、C1 或核心。通用搜索投影、Obsidian、子库和
  输出 builder 仍分别属于 P6.2/P6.3。

以下实现选择仍未锁定，不是产品意图空缺：

- 衰减和基于真实使用数据的调权建议算法；
- 窄 UI 的具体框架和移动端/桌面端优先级；
- 首个内容浮现界面的节奏和排序细节；
- 首批正式启用的输出 builder。

这些选择必须由真实资料、真实调用者和对应测试推动，不能反向改变本文已确定的语义。
