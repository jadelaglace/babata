# Babata 来源工具调研与路线决策

## 1. 文档职责

本文是 P2 的来源工具调查证据，不是工具愿望清单。它回答：面对每个网站、平台和
运营商，现有工具到底能不能把属于用户的资料拿回来；需要用户最少提供什么；Agent
能完成到哪一步；Babata 还需不需要写代码。

未经调查和实际验证，不得用 provider 空文件、接口设计、导入格式或手工导出代替
来源路线。本文完成前，P2 的工具规划门不通过。

## 2. 强制调查顺序

```text
官方 API / 官方客户端能力
-> 可直接调用的 CLI、SDK、MCP 或 Agent 工具
-> 浏览器扩展、用户脚本和可复用登录会话的浏览器自动化
-> 仍维护的成熟开源工具
-> 只补缺口的窄适配器
-> 手工导出、PDF、复制、截图、录屏（最后回退）
```

搜索到项目名称不算调研完成。每条候选路线要检查官方文档/仓库、维护状态、许可、
认证方式，并在安全和权限允许时实际安装、登录、列候选或读取一个用户授权样本。

## 3. 每个来源必须回答

| 字段 | 必须记录的内容 |
| --- | --- |
| 用户要拿回什么 | 收藏、正文、对话、媒体、附件、层级、时间、作者、原链接等 |
| 官方路线 | API、数据导出、客户端备份或开放平台能力 |
| 成熟工具 | CLI、SDK、MCP、Agent 工具、扩展、用户脚本、开源项目 |
| Agent 可调用性 | 命令/API/浏览器会话是否能由 Agent 稳定触发 |
| 最小用户提供 | OAuth、token、cookie、已登录浏览器、扫码、导出件或本地备份 |
| 候选与选择 | 能否先列出标题/层级/数量，让用户明确选择范围 |
| 取得内容 | 正文、原文件、附件、媒体、元数据和上下文的实际覆盖 |
| 重收集 | 是否有稳定 ID、更新时间、增量、变化和删除判断 |
| 实际证据 | 安装/调用命令、授权样本结果、官方文档和项目链接 |
| 限制与风险 | 权限、频率、登录、平台规则、失效风险和信息损失 |
| 决策 | 直接用、薄包装、窄适配、最后回退或暂不支持 |

## 5. 路线决策规则

- **直接用**：现有工具已覆盖真实需要，Babata 只负责调用并接收结果。
- **薄包装**：工具可用，只需补授权启动、范围选择或输出规范化。
- **窄适配**：现有工具缺少一个已证明的重要能力，只补该缺口。
- **最后回退**：所有正常工具路线确实不可用，才要求手工导出/复制/截图/录屏。
- **暂不支持**：授权、合法性、稳定性或完整性无法接受时，如实停止。

手工路线不能因为容易写 fixture 或离线测试，就被提升为正常产品路线。
